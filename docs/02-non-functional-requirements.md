# نظام نقطة البيع + المخزون + المحاسبة - المتطلبات غير الوظيفية

## الأداء
- زمن استجابة نقطة البيع أقل من ثانية واحدة في السيناريوهات المعتادة
- دعم مستخدمين متزامنين
- تحديث مخزون لحظي بدون تأخير ملحوظ

ملاحظات تنفيذ:
- فهارس (indexes) واستعلامات محسّنة
- استخدام caching عند الحاجة وبشكل آمن
- فصل المهام الثقيلة (تقارير) إلى عمليات خلفية

## الاعتمادية
- هدف 99% توافر (Uptime)
- اتساق البيانات إلزامي (المخزون والمحاسبة لا يجب أن يتباعدا)
- نسخ احتياطي يومي وخطة تعافي من الكوارث

## قابلية التوسع
- عدة فروع/مخازن
- آلاف المنتجات مع نمو سجل العمليات

ملاحظات تنفيذ:
- فصل الخدمات (modular services) وحدود واضحة
- API عديم الحالة (stateless) لتسهيل التوسع الأفقي عند الحاجة

## الأمان
- تشفير كلمات المرور: `bcrypt`
- صلاحيات حسب الدور (RBAC) على كل الـ APIs
- حماية الجلسات/التوكنات
- سجلات تدقيق للأحداث الحرجة
- المدفوعات عبر HTTPS والتحقق من عمليات الدفع

## قابلية الاستخدام
- واجهة POS سريعة ومناسبة للشاشة اللمسية ولوحة المفاتيح
- تقليل النقرات مع اختصارات منطقية
- تصميم Responsive للهواتف/الأجهزة اللوحية

## قابلية الصيانة
- معمارية وحدات:
  - وحدة POS
  - وحدة المخزون
  - وحدة المحاسبة
  - وحدة التقارير
  - وحدة الإشعارات
  - وحدة المستخدمين/المصادقة

## سلامة البيانات
- منع المخزون من النزول إلى قيم سالبة
- استخدام معاملات ACID
- التراجع (rollback) عند الفشل
- تحديث المخزون والمحاسبة يجب أن يكونان ذريّين (atomic) ضمن العملية الواحدة

## العمل دون اتصال (Offline) لـ POS
- POS يعمل بدون إنترنت ثم تتم المزامنة لاحقًا

ملاحظات تنفيذ:
- Service Workers
- IndexedDB لتخزين البيانات مؤقتًا
- إعادة تشغيل (replay) العمليات المعلقة عند عودة الاتصال

## التوطين (Localization)
- دعم لغتين: العربية/الإنجليزية (اختياري للتوسع)
- دعم عملات متعددة
- i18n في React/Next.js
