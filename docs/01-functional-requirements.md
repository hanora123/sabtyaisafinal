# نظام نقطة البيع + المخزون + المحاسبة - المتطلبات الوظيفية

## 1.1 إدارة المستخدمين والصلاحيات
المطلوب:
- إنشاء المستخدمين بالأدوار: `Admin` و`Cashier` و`Accountant`

الصلاحيات حسب الدور:
- `Admin`: صلاحيات كاملة
- `Cashier`: نقطة البيع + صلاحيات مخزون محدودة
- `Accountant`: تقارير + قسم مالي فقط

المصادقة والجلسات:
- نظام تسجيل دخول/خروج متكامل يعتمد على ملفات تعريف الارتباط المؤمنة (Secure JWT Cookies).
- دعم **أدوار متعددة** لكل مستخدم، مما يسمح بمرونة عالية في الصلاحيات.
- **توجيه تلقائي ذكي**:
  - `Admin`: يتم توجيهه تلقائيًا إلى لوحة الإدارة (`/admin`).
  - `Cashier`: يتم توجيهه تلقائيًا إلى نقطة البيع (`/pos`).
- التحقق من الصلاحيات (RBAC) في كل من الواجهة الأمامية (Front-end) والخلفية (Back-end).
- 2FA اختياري عند تفعيله.
سجلات التدقيق (لازم تكون غير قابلة للتعديل وبناء أحداث):
- إجراءات المستخدم: إنشاء/تعديل/حذف
- أحداث المصادقة: دخول/خروج
- أحداث POS: إتمام البيع (ورد/استرجاع إن وجد)

جداول قاعدة بيانات مقترحة:
- `Users`
- `Roles`
- `UserRoles`
- `AuditLogs`

الأمان:
- تجزئة كلمات المرور: `bcrypt`
- تطبيق مبدأ أقل صلاحيات على كل الـ APIs

## 1.2 إدارة المنتجات (CRUD)
متطلبات المنتج:
- الاسم
- SKU / باركود
- التصنيف
- سعر التكلفة
- سعر البيع
- المورد

عمليات CRUD:
- إنشاء/تحديث/حذف المنتجات
- تعطيل المنتجات بدل الحذف النهائي (لحفظ التاريخ)

استيراد/تصدير جماعي:
- CSV/Excel

جداول مقترحة:
- `Products`
- `Categories`
- `Suppliers`

الواجهة:
- لوحة إدارة React / Next.js

## 1.3 إدارة المخزون (مخزون متعدد)
المطلوب:
- عدة مخازن (مثل: المستودع، الفرع A، …)
- تتبع:
  - الكمية المتوفرة
  - الكمية المحجوزة (لعمليات بيع/طلبات معلقة)

عمليات المخزون:
- إضافة
- إزالة
- تعديل (تصحيح)
- تحويل بين المخازن

تنبيهات نقص المخزون:
- عند انخفاض المتاح عن الحد الأدنى للمنتج

معادلة المخزون:
- `Inventory = Opening + Purchases - Sales - Transfers`

جداول مقترحة:
- `Inventories`
- `Stock`
- `StockMovements`

تحديثات لحظية:
- WebSockets لدفع تحديثات المخزون إلى شاشات POS/الإدارة

## 1.4 POS / نظام الإتمام (Checkout)
متطلبات الإتمام:
- البحث عن المنتجات بالاسم أو الباركود
- إضافة المنتجات إلى السلة وتعديل الكميات
- تطبيق الخصومات:
  - مبلغ ثابت
  - نسبة مئوية
- قبول مدفوعات متعددة:
  - كاش
  - بطاقة
  - محفظة
- إصدار الإيصال:
  - طباعة
  - PDF رقمي

الإضافة السريعة (Quick Add):
- إضافة منتج أثناء البيع بالباركود/SKU
- قاعدة اختيارية: إذا المنتج غير موجود يتم إنشاؤه تلقائيًا
- ثم إضافته للسلة

لحظية:
- تحديث توفر المخزون داخل السلة عبر WebSocket عند تغير المخزون

جداول مقترحة:
- `Sales`
- `SaleItems`
- `Payments`

## 1.5 إدارة المبيعات
التسجيل:
- حفظ كل عملية بيع (العناصر، الكمية، الإجمالي، طريقة الدفع)

المرتجعات/استرداد المبالغ:
- كامل أو جزئي
- تصحيح المخزون وفقًا لذلك

ملخص يومي:
- تجميع المبيعات حسب اليوم/فواصل زمنية

جداول مقترحة:
- `Sales`
- `SaleItems`
- `Payments`

قاعدة عمل:
- بعد إنشاء البيع يتم تحديث المخزون وتوليد القيود المحاسبية تلقائيًا

## 1.6 إدارة الموردين
المطلوب:
- CRUD للموردين
- ربط المنتجات بالمورد
- تسجيل المشتريات من الموردين
- تتبع المبالغ غير المسددة (Payables)

جداول مقترحة:
- `Suppliers`
- `Purchases`
- `PurchaseItems`

## 1.7 المشتريات (Stock In)
أوامر الشراء:
- إنشاء أوامر شراء
- إضافة الأصناف والكميات
- تحديث المخزون تلقائيًا عند إدخال البضاعة
- تتبع التكلفة/الكمية/التاريخ

منطق القيود (قصد Double-Entry عادةً):
- إنشاء `StockMovement`
- إنشاء `JournalEntry`:
  - مدين: المخزون (أو أصل)
  - دائن: مستحقات المورد

## 1.8 المحاسبة والمالية (الأساس)
المطلوب:
- محاسبة بقيد مزدوج (Double-entry)
- قيود تلقائية لكل المبيعات/المشتريات
- دليل الحسابات: الأصول، الخصوم، حقوق الملكية، الإيرادات، المصروفات
- تتبع التدفقات النقدية (دخول/خروج) ورصيد يومي
- إدارة المصروفات: إيجار، رواتب، خدمات…
- حساب الربح:
  - الإجمالي = المبيعات - COGS
  - الصافي = الإجمالي - المصروفات
- تقارير مالية: الأرباح والخسائر، الميزانية، التدفقات النقدية، تقييم المخزون

جداول مقترحة:
- `Accounts`
- `JournalEntries`
- `JournalEntryLines`

## 1.9 التقارير والتحليلات
متطلبات:
- تقارير المبيعات (يومي/أسبوعي/شهري)
- أفضل المنتجات + الربح لكل منتج
- تقارير حركة المخزون
- لوحة تحكم: الإيرادات، الأرباح، تنبيهات المخزون

لحظية:
- تحديثات عبر WebSocket/SignalR

## 1.10 الإشعارات
متطلبات:
